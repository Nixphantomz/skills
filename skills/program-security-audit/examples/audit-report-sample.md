## Threat Model

### Assets at Risk
- User staked SOL (held in program-controlled vault PDA)
- Accumulated staking rewards (held in reward pool account)
- Admin authority over reward rate and emergency pause

### Entry Points
| Instruction | Caller | Assets Affected |
|---|---|---|
| `initialize` | Admin only | Creates vault and config |
| `stake` | Any user | Deposits SOL to vault |
| `unstake` | Position owner | Withdraws SOL from vault |
| `claim_rewards` | Position owner | Transfers rewards to user |
| `update_rate` | Admin only | Changes reward rate |
| `emergency_withdraw` | Admin only | Drains vault to admin |

### Threat Actors
- **External attacker**: No program interaction history, no authority
- **Malicious staker**: Valid user who attempts to exploit logic
- **Rogue admin**: Admin key compromised or malicious upgrade

## Findings

### [CRITICAL-01] Missing Ownership Check on Vault Account

**Location**: `lib.rs`, instruction `unstake`, line 87

**Description**: The `unstake` instruction accepts a `vault` account as `AccountInfo` without verifying it is owned by this program. An attacker can construct a fake vault account with spoofed balance data, causing the program to transfer more SOL than the user actually staked.

**Vulnerable Code**:
```rust
pub fn unstake(ctx: Context<Unstake>, amount: u64) -> Result<()> {
    let vault = &ctx.accounts.vault; // AccountInfo, no ownership check
    let recorded = vault.data.borrow()[0..8]; // reads from attacker-controlled data
    // ...transfer based on recorded balance
}
```

**Impact**: An attacker can drain the entire vault in a single transaction.

**Recommendation**:
```rust
// Option A: Use Anchor's Account<> type (auto-checks ownership)
pub vault: Account<'info, Vault>,

// Option B: Manual check for native programs
require!(
    vault.owner == ctx.program_id,
    ErrorCode::InvalidVaultOwner
);
```

---

### [HIGH-01] Unchecked Arithmetic in Reward Calculation

**Location**: `rewards.rs`, function `calculate_rewards`, line 34

**Description**: The reward calculation multiplies `staked_amount` by `rate` and `time_elapsed` without overflow protection. In release builds, this silently wraps, potentially awarding a user far fewer (or zero) rewards than owed.

**Vulnerable Code**:
```rust
let rewards = staked_amount * rate * time_elapsed; // overflows silently in release
```

**Impact**: Users receive incorrect reward amounts. Could be exploited by staking for a precise duration to trigger wrap-around and claim excess rewards.

**Recommendation**:
```rust
let rewards = (staked_amount as u128)
    .checked_mul(rate as u128)
    .ok_or(ErrorCode::Overflow)?
    .checked_mul(time_elapsed as u128)
    .ok_or(ErrorCode::Overflow)?
    .checked_div(REWARD_PRECISION as u128)
    .ok_or(ErrorCode::DivisionByZero)? as u64;
```

### [HIGH-02] Single-Step Authority Transfer

**Location**: `admin.rs`, instruction `transfer_authority`, line 12

**Description**: The admin can transfer authority to any pubkey in a single step. If an incorrect address is supplied (typo or front-run), the program loses its admin permanently.

**Vulnerable Code**:
```rust
pub fn transfer_authority(ctx: Context<TransferAuthority>, new_admin: Pubkey) -> Result<()> {
    ctx.accounts.config.admin = new_admin; // immediate, irreversible
    Ok(())
}
```

**Impact**: Permanent loss of admin access if wrong pubkey supplied.

**Recommendation**: Implement two-step transfer (propose + accept). See `docs/access-control.md` Pattern 2.

---

### [MEDIUM-01] No Deadline on `unstake` Instruction

**Location**: `lib.rs`, instruction `unstake`

**Description**: The `unstake` instruction has no expiry. A user's signed transaction could sit in the mempool and execute at an unfavorable time, potentially after a reward rate change the user was unaware of.

**Recommendation**:
```rust
pub fn unstake(ctx: Context<Unstake>, amount: u64, deadline: i64) -> Result<()> {
    let clock = Clock::get()?;
    require!(clock.unix_timestamp <= deadline, ErrorCode::TransactionExpired);
    // ...
}
```

---

### [LOW-01] `init_if_needed` Used for Position Account

**Location**: `lib.rs`, instruction `stake`, line 45

**Description**: Using `init_if_needed` allows a position to be reinitialized. While the current logic checks `position.is_initialized`, this relies on manual state rather than Anchor's discriminator protection.

**Recommendation**: Use `init` constraint. If re-staking to an existing position is intentional, use a separate `add_stake` instruction.

---

### [INFO-01] No Events Emitted

**Description**: No instructions emit events via `emit!()`. This makes off-chain monitoring and indexing significantly harder.

**Recommendation**: Add events for all state-changing instructions:
```rust
#[event]
pub struct StakeEvent {
    pub user: Pubkey,
    pub amount: u64,
    pub timestamp: i64,
}
// In instruction: emit!(StakeEvent { user: ..., amount, timestamp });
```

---

### [INFO-02] Missing NatSpec / Documentation Comments

**Description**: Instruction functions lack `///` doc comments explaining parameters, expected account state, and error conditions.
